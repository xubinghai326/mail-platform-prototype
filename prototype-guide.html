Prototype README

这套原型是为了直观展示
邮件发放平台如何守住高风险操作

原型围绕一个核心问题展开：如何在运营效率与操作安全之间取得平衡，避免“把 10000 金币误发成 10000 钻石”这类高危事故。 因此页面重点展示的不是视觉设计本身，而是“风险如何被识别、流程如何被约束、异常如何被止损、Agent 如何在安全边界内提效”。

关键交互怎么理解

下面这些是最值得在演示时点出来的交互。

Interaction Notes
高风险数值熔断

当奖励数值明显偏离历史基线，例如把基础货币误配成高价值钻石时，系统会直接显示高危状态，并锁定提交审批。

范围偏差告警

如果目标分群从“异常玩家”漂移成“全服玩家”，系统会提示目标人数与画像偏差，并要求更严格的灰度验证。

重新分析风险

点击后会重新计算数值、范围和高价值资产暴露，并给出分析中状态、完成时间和最新结论。

不可变快照与 Hash

进入审批后系统会冻结配置并生成唯一 Hash，防止“审批的是 A，发出去的是 B”。

正式服灰度与小流量试发

不是只验证“能不能领”，而是验证目标分群、不同样本和真实小流量放出后的行为表现。

Agent 区块怎么看

它不是自动发奖励，而是负责分析和辅助。

Agent Boundaries
能做什么

监控异常、汇总样本、生成建议、草拟公告、补全工单、沉淀复盘材料。

不能做什么

不能直接替代人工审批，不能直接拥有全服发放权，也不能自行决定高风险资产追回。

演示时可以这样讲

如果你向别人汇报，可以按这个顺序快速说明设计价值。

Demo Script
A
先讲问题背景

上一轮事故本质上是高价值配置误发，因此平台目标不是只发得快，而是先把高危操作拦住。

B
再讲主链路

从建单、风险识别、审批、灰度到快照冻结和发前预检，说明风险如何被逐层约束。

C
最后讲异常处置

即使前面失守，系统也要支持快速熔断、分层止损、公告节奏和复盘沉淀。

一句话总结这套原型

适合放在汇报结尾，帮助听的人快速记住重点。

Wrap-up

高风险强约束，关键决策人工拍板，重复事务 Agent 提效。

这套原型的价值不在于把邮件“更快地发出去”，而在于把“发前防错、发中控制、发后止损”做成一个可落地、可解释、可演示的系统闭环。
